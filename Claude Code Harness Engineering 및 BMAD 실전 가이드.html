PART 1 개념 이해

1. Harness Engineering이란 무엇인가

OpenAI의 정의
OpenAI의 공식 글("Harness engineering: leveraging Codex in an agent-first world")에서 가장 명확하게 정의합니다:

"Our most difficult challenges now center on designing environments, feedback loops, and control systems that help agents accomplish our goal: build and maintain complex, reliable software at scale."
(출처: https://openai.com/index/harness-engineering/)

즉, 코드를 직접 작성하는 것이 아니라 에이전트가 올바르게 작동하도록 환경, 피드백 루프, 제어 시스템을 설계하는 것이 핵심입니다. OpenAI 팀은 수작업 코드 없이 5개월 만에 100만 줄 이상의 프로덕션 코드베이스를 구축했으며, 그 과정에서 scaffolding(비계), tooling(도구), abstraction(추상화) 설계가 엔지니어의 주된 책임이 되었습니다.

Martin Fowler의 해석 — 3가지 레이어
Martin Fowler는 harness를 세 가지 범주로 분류합니다:
(출처: https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)

레이어 내용 예시
Context Engineering 에이전트에게 필요한 지식을 구조적으로 제공 CLAUDE.md, docs/, 관찰 가능성 데이터
Architectural Constraints LLM + 결정론적 도구로 구조를 강제 커스텀 린터, 구조 테스트, 레이어 의존성
Garbage Collection 엔트로피와 드리프트를 주기적으로 제거 문서 정합성 검사 agent, 리팩터링 PR 자동 생성

Fowler의 핵심 주장: "AI를 신뢰 가능하게 하려면 런타임과 구조를 제약해야 한다." 자유도는 줄어들지만, 유지보수 가능성과 신뢰성은 올라갑니다.

HumanLayer의 정리 — Harness의 6가지 구성요소
(출처: https://www.humanlayer.dev/blog/skill-issue-harness-engineering-for-coding-agents)

  1. CLAUDE.md / AGENTS.md: 에이전트 시스템 프롬프트에 결정론적으로 주입되는 프로젝트 지침
  2. MCP Servers: 에이전트의 툴킷을 외부 서비스로 확장
  3. Skills: 필요할 때만 지식/도구를 로드하는 점진적 공개(progressive disclosure) 메커니즘
  4. Sub-Agents: 메인 컨텍스트를 오염시키지 않고 독립적으로 작업을 처리하는 전문 에이전트
  5. Hooks: 에이전트 lifecycle의 특정 지점에서 자동 실행되는 커스텀 명령어
  6. 검증 메커니즘(Back-Pressure): 에이전트가 자신의 작업을 직접 검증하는 루프

HumanLayer의 핵심 원칙: "검증은 context-efficient해야 한다." 성공 로그를 길게 컨텍스트에 밀어 넣으면 instruction budget이 감소합니다. 성공 시 조용하게 통과시키고, 실패 시에만 에러 메시지를 노출하는 구조가 효과적입니다.

세 가지 엔지니어링의 차이

구분 초점 예시
Prompt Engineering 단일 요청의 품질 개선 "코드를 짜줘" → "TypeScript로, 에러 처리 포함해서 짜줘"
Context Engineering 에이전트에게 전달되는 정보의 품질과 양 제어 CLAUDE.md, docs/, 메모리 관리
Harness Engineering 에이전트가 작동하는 전체 환경과 제어 시스템 설계 CLAUDE.md + Rules + Hooks + Subagents + Skills + 검증 루프

2. BMAD 사용자라면 Harness를 어디에 위치시켜야 하나

BMAD Method와 Harness Engineering은 서로 다른 레이어에서 작동합니다.

BMAD = 사고와 산출물의 프로세스 레이어
BMAD는 "무엇을 만들 것인가"와 "어떻게 구조화할 것인가"를 체계적으로 정의합니다: PRD.md, architecture.md, epics/, stories, sprint-status.yaml

Harness = 실행 품질과 안전장치
Harness는 "BMAD가 만든 산출물을 Claude가 일관되게 따르고, 규칙을 지키며, 위험한 행동을 하지 않도록 만드는 장치"입니다.

BMAD 설치 결과 확인
(출처: https://docs.bmad-method.org/how-to/install-bmad/) 

_bmad/              # 에이전트, 워크플로우, 태스크, 설정
_bmad-output/       # 생성된 산출물
.claude/skills/
  ├── bmad-help/
  ├── bmad-create-prd/
  ├── bmad-dev/
  └── ...

핵심 원칙: BMAD skills는 그대로 유지하고, 여기에 별도로 CLAUDE.md, Rules, Hooks, Subagents를 추가합니다. BMAD를 대체하는 것이 아니라 BMAD 위에 Harness를 얹는 구조입니다.

3. 처음 도입할 때의 권장 순서

  1. BMAD 설치 + 프로젝트 문서 정리 — 기반 산출물 확보
  2. 루트 CLAUDE.md 작성 — 에이전트의 기본 행동 원칙 주입
  3. .claude/rules/로 규칙 분리 — 도메인별 규칙 모듈화
  4. .claude/settings.json에 최소 Hook 적용 — 위험 차단 + 자동 검증
  5. 코드리뷰/디버깅용 Subagent 2개 만들기 — 컨텍스트 분리 시작
  6. Plan Mode + Git Worktree 운영 습관 붙이기 — 병렬 작업 효율화
  7. Lint/Test/Build를 CI와 Hook으로 기계적으로 연결 — 검증 자동화
  8. 필요할 때만 Skills, MCP, Background Agent 확장 — 과잉 설정 방지

PART 3 단계별 설정 방법

Step 1: BMAD Method 먼저 설치

(출처: BMAD Install Docs, BMAD Getting Started)

신규 프로젝트

# 프로젝트 디렉터리로 이동
cd your-project

# BMAD 설치 (인터랙티브)
npx bmad-method install
# 설치 중 선택: AI Tool → Claude Code, Module → BMad Method

설치 후 생성되는 디렉터리:

_bmad/              # 에이전트, 워크플로우, 태스크, 설정 (수정 금지)
_bmad-output/       # 비어 있음 (산출물 저장 예정)
.claude/skills/
  bmad-help/SKILL.md
  bmad-create-prd/SKILL.md
  bmad-dev/SKILL.md
  ... (선택한 모듈에 따라 다수)

설치 확인

# Claude Code에서 실행
bmad-help

기존 프로젝트에 적용

# 코드베이스 분석 후 project-context.md 자동 생성
bmad-generate-project-context
# 생성 위치: _bmad-output/project-context.md

이 파일은 기술 스택, 코드 구조, 네이밍 컨벤션 등을 자동으로 분석합니다.

Step 2: 루트 CLAUDE.md 작성

(출처: Anthropic Memory Docs)

Anthropic 권장 원칙

  • 파일당 200줄 이하 유지
  • 구체적이고 검증 가능한 문장으로 작성 ("코드를 잘 짜라" ❌ → "2-space 들여쓰기를 사용하라" ✅)
  • Markdown 헤더와 불릿으로 구조화
  • @path 문법으로 외부 파일 참조 (import)
  • 모든 내용을 담는 백과사전이 아니라, "어디를 봐야 하는지 알려주는 목차"

메모리 로드 우선순위

우선순위위치설명
1 (최고)Managed policy조직 관리자가 설정한 정책
2~/.claude/CLAUDE.md사용자 전역 설정
3./CLAUDE.md 또는 ./.claude/CLAUDE.md프로젝트 설정
4 (최저)서브디렉터리 CLAUDE.md해당 디렉터리 접근 시 on-demand 로드

@path import 문법

# 다른 파일의 내용을 현재 컨텍스트에 포함
@docs/10-architecture/system-overview.md
@AGENTS.md
@_bmad-output/planning-artifacts/PRD.md

실전 CLAUDE.md 템플릿 전문

# CLAUDE.md

## Project Overview
This is [프로젝트명]. [한 줄 설명].

- **Product Requirements**: @_bmad-output/planning-artifacts/PRD.md
- **Architecture**: @docs/10-architecture/system-overview.md
- **Tech Stack**: @docs/10-architecture/tech-stack.md
- **Current Sprint**: @_bmad-output/implementation-artifacts/sprint-status.yaml
- **Project Context**: @_bmad-output/project-context.md

## Operating Mode
1. Always start in Plan Mode (`Shift+Tab`) for tasks affecting more than 2 files.
2. After planning, confirm with me before implementing.
3. For new features, check if a corresponding story exists in `_bmad-output/`.
4. Do not modify `_bmad/` directory.

## Build, Test & Quality
```bash
npm run dev          # 개발 서버
npm run build        # 빌드
npm run test         # 전체 테스트
npm run lint         # 린트
npm run typecheck    # 타입체크
```
- Run `npm run lint && npm run typecheck` before completing any implementation task.
- Tests must pass. Do not skip or comment out failing tests.

## Product & Delivery Constraints
- All user-facing strings must be in `src/i18n/`.
- Never hardcode API keys or secrets. Use environment variables only.
- Pricing and billing logic must be reviewed before committing.
- @docs/00-product/vision.md

## Implementation Style
- Language: TypeScript (strict mode)
- Indentation: 2 spaces
- File naming: kebab-case for files, PascalCase for components
- Function naming: camelCase
- Error handling: always use typed errors, never `any`
- @.claude/rules/code-style.md

## BMAD Integration
- BMAD skills are in `.claude/skills/bmad-*/` — do not modify these files.
- Before implementing a story, read the full story file in `_bmad-output/`.
- After completing implementation, update `sprint-status.yaml`.
- Planning artifacts in `_bmad-output/planning-artifacts/` are the source of truth.

## Reporting
- When you complete a task, summarize: what you changed, what you ran, what passed.
- If blocked, report: what you tried, what failed, what's needed.
- Do not hallucinate test results. Run them and report actual output.

Step 3: .claude/rules/로 규칙 분리

(출처: Anthropic Memory Docs)

큰 프로젝트에서 CLAUDE.md 하나에 모든 규칙을 넣으면 컨텍스트가 낭비되고 내용 관리가 어렵습니다. .claude/rules/를 사용하면 규칙을 모듈별로 분리할 수 있습니다.

자동 로드 규칙

위치범위paths: 지원
~/.claude/rules/*.md사용자 전역 (모든 프로젝트)✅
.claude/rules/*.md현재 프로젝트✅
paths: frontmatter 없는 파일세션 시작 시 항상 로드—
paths: frontmatter 있는 파일해당 glob 패턴 파일 작업 시 on-demand 로드✅

code-style.md

---
description: 전체 프로젝트 코드 스타일 규칙
---

# Code Style Rules

## General
- Use TypeScript strict mode at all times.
- No `any` type. Use `unknown` and narrow with type guards.
- No `console.log` in production code. Use the project logger at `src/lib/logger.ts`.
- Prefer `const` over `let`. Never use `var`.

## Naming
- Files: `kebab-case.ts`
- Components: `PascalCase.tsx`
- Functions and variables: `camelCase`
- Constants: `SCREAMING_SNAKE_CASE`
- Interfaces: prefix with `I` is forbidden. Use plain names.

## Imports
- Sort imports: external packages → internal aliases → relative paths.
- No default exports from utility files. Named exports only.

## Comments
- Write comments for "why", not "what".
- JSDoc for all public functions and types.

# Testing Rules

## Framework
- Unit/Integration: Vitest
- E2E: Playwright
- Test files: `*.test.ts` or `*.spec.ts` co-located with source files.

## Coverage Requirements
- New business logic: minimum 80% branch coverage.
- Utility functions: 100% coverage.
- UI components: snapshot + key interaction tests.

## Rules
- Never skip tests with `it.skip` or `test.todo` without a linked issue.
- Mock external APIs. Never call real endpoints in unit tests.
- Use `describe` blocks to group related tests.
- Test file must import the subject under test, not its internals.

## Running Tests
```
npm run test           # 전체 실행
npm run test:unit      # 단위 테스트만
npm run test:e2e       # E2E 테스트만
npm run test -- --coverage  # 커버리지 포함
```

# Security Rules

## Secrets & Environment
- NEVER hardcode API keys, passwords, or tokens in source code.
- Use `.env.local` for local secrets (already in .gitignore).
- Reference secrets via `process.env.VAR_NAME` only.
- Validate all environment variables at startup using a schema (e.g., zod).

## Input Validation
- Validate ALL user inputs at the API boundary using zod or equivalent.
- Sanitize HTML output. Never use `dangerouslySetInnerHTML` without explicit sanitization.
- Parameterize all database queries. No string concatenation for SQL.

## Authentication & Authorization
- Never bypass authentication middleware.
- Check authorization at the service layer, not just the route layer.
- Session tokens must be httpOnly, Secure, SameSite=Strict.

## Dependencies
- Do not add new dependencies without reviewing for known vulnerabilities.
- Run `npm audit` before any release.

# Backend API Rules

## Request/Response
- All API responses must follow the standard envelope:
  { "data": ..., "error": null }
  { "data": null, "error": { "code": "...", "message": "..." } }
- Use HTTP status codes correctly. 200 for success, 4xx for client errors, 5xx for server errors.
- Never expose internal error messages to clients.

## Middleware Order
1. Authentication
2. Rate limiting
3. Input validation
4. Business logic
5. Response formatting

## Database
- Use transactions for multi-table writes.
- Index foreign keys and frequently queried columns.
- Never run migrations without a rollback plan.

block-rm.sh — 위험 명령어 차단 스크립트

스크립트 작성 후 반드시 실행 권한을 부여하세요: chmod +x .claude/hooks/block-rm.sh

#!/bin/bash
# .claude/hooks/block-rm.sh

INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // ""')

DANGEROUS_PATTERNS=(
  "rm -rf"
  "rm -fr"
  "sudo rm"
  "sudo dd"
  "> /etc/"
  "> /usr/"
  "chmod 777"
  "chmod -R 777"
  "truncate /"
)

for PATTERN in "${DANGEROUS_PATTERNS[@]}"; do
  if echo "$COMMAND" | grep -qF "$PATTERN"; then
    jq -n \
      --arg reason "Blocked by harness: dangerous command pattern '$PATTERN' detected in: $COMMAND" \
      '{
        hookSpecificOutput: {
          hookEventName: "PreToolUse",
          permissionDecision: "deny",
          permissionDecisionReason: $reason
        }
      }'
    exit 2
  fi
done

exit 0

run-checks.sh — 파일 수정 후 자동 검증 스크립트

HumanLayer의 "context-efficient verification" 패턴: 성공 시 아무 출력 없음, 실패 시에만 stderr로 에러 보고.

#!/bin/bash
# .claude/hooks/run-checks.sh
# 파일 수정 후 lint + typecheck 자동 실행
# 성공 시: 아무 출력 없음 (context-efficient)
# 실패 시: stderr로 에러만 보고

set -euo pipefail

PROJECT_DIR="$CLAUDE_PROJECT_DIR"
cd "$PROJECT_DIR"

# stdin에서 JSON 입력 읽기
INPUT=$(cat)
MODIFIED_FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // ""')

# 소스 파일이 아니면 검증 생략
if [[ -z "$MODIFIED_FILE" ]] || \
   [[ "$MODIFIED_FILE" != *.ts ]] && \
   [[ "$MODIFIED_FILE" != *.tsx ]] && \
   [[ "$MODIFIED_FILE" != *.js ]] && \
   [[ "$MODIFIED_FILE" != *.jsx ]]; then
  exit 0
fi

# package.json 없으면 생략
if [[ ! -f "package.json" ]]; then
  exit 0
fi

# 린트 실행 (실패 시 에러 출력)
LINT_OUTPUT=$(npm run lint --silent 2>&1) || {
  echo "❌ LINT FAILED — fix before proceeding:" >&2
  echo "$LINT_OUTPUT" >&2
  exit 2
}

# 타입체크 실행 (실패 시 에러 출력)
TYPE_OUTPUT=$(npm run typecheck --silent 2>&1) || {
  echo "❌ TYPECHECK FAILED — fix before proceeding:" >&2
  echo "$TYPE_OUTPUT" >&2
  exit 2
}

# 모든 검증 통과: 조용히 종료
exit 0

1. **Gather context**:
   - Read error messages and stack traces carefully
   - Read relevant source files
   - Check recent git changes: `git log --oneline -10` and `git diff HEAD~1`
2. **Reproduce the issue**:
   - Identify the minimal reproduction path
   - Run relevant tests: `npm run test -- <test-file>` if applicable
3. **Trace the root cause**:
   - Follow the call stack from error origin
   - Check environment variables and configuration
   - Identify whether it's a code bug, config issue, or dependency problem
4. **Propose and apply fix**:
   - Describe the root cause clearly
   - Apply the minimal fix
   - Run the failing test again to confirm resolution
5. **Report**:
   - Root cause summary
   - Files changed
   - Test result before/after

Always run lint and typecheck after applying a fix:
npm run lint && npm run typecheck

validate-readonly-query.sh

#!/bin/bash
# .claude/hooks/validate-readonly-query.sh
# db-reader subagent 전용: SQL 쓰기 명령어 차단

INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // ""')

WRITE_PATTERNS=("INSERT" "UPDATE" "DELETE" "DROP" "TRUNCATE" "ALTER" "CREATE" "GRANT")
UPPER_CMD=$(echo "$COMMAND" | tr '[:lower:]' '[:upper:]')

for PATTERN in "${WRITE_PATTERNS[@]}"; do
  if echo "$UPPER_CMD" | grep -qw "$PATTERN"; then
    jq -n \
      --arg reason "db-reader agent is read-only. '$PATTERN' statements are not allowed." \
      '{
        hookSpecificOutput: {
          hookEventName: "PreToolUse",
          permissionDecision: "deny",
          permissionDecisionReason: $reason
        }
      }'
    exit 2
  fi
done

exit 0

15. 기계적 검증 강화

(출처: OpenAI Harness Engineering, Boris Cherny 인터뷰)

검증 계층 구조

계층타이밍도구
로컬 Hook즉시 (파일 수정 직후)lint, typecheck
Subagent Review구현 완료 후code-reviewer, release-checker
CIPR 생성 / 머지 전전체 테스트 suite, 보안 스캔

OpenAI의 기계적 강제 패턴

  • 커스텀 린터: 반복적인 코드 리뷰 코멘트가 생기면, 그 규칙을 린터 룰로 승격합니다.
  • 구조 테스트: 레이어 의존성 방향, 금지된 import 패턴 등을 자동 테스트로 검사합니다.
  • 문서 최신성 CI 잡: docs/가 실제 코드와 일치하는지 주기적으로 검사합니다.

Boris Cherny의 검증 원칙: PR마다 Claude Code가 1차 리뷰(~80% 버그 탐지), 타입체커/린터/빌드가 뒤따르고, 인간 엔지니어가 최종 리뷰합니다.

16. Agentic Search > Local RAG

Boris Cherny 인터뷰에서의 실용적 통찰: 라이브 레포에서는 로컬 벡터DB/RAG보다 agentic search(glob/grep)이 더 우수합니다. 인덱스 드리프트와 권한 문제가 없기 때문입니다.

코드베이스 검색 전략 권장 순서

우선순위전략이유
1CLAUDE.md + rules로 구조 명시가장 저렴하고 결정론적
2docs/ 정리 + @import정확한 컨텍스트 직접 제공
3glob/grep 기반 agentic search드리프트 없이 실시간 검색
4RAG/벡터DB대규모 레거시 코드베이스 한정

17. MCP 운영 원칙

(출처: HumanLayer)

HumanLayer의 경고: MCP 서버가 많아질수록 에이전트의 instruction budget(시스템 프롬프트 용량)이 감소합니다. 툴이 많으면 오히려 성능이 저하됩니다.

MCP 도입 우선순위

우선순위MCP이유
1순위GitHub MCPPR, 이슈, 코드 리뷰 자동화
1순위Playwright MCPE2E 테스트, UI 디버깅
2순위DB read-only MCP데이터 조회 (쓰기 차단 필수)
2순위로그/에러 추적 MCPSentry, Datadog 등
후순위Slack, Notion, Gmail도입 효과 먼저 검증 후 추가

Subagent의 frontmatter에서 mcpServers를 명시적으로 지정하면, 메인 세션의 MCP 권한이 subagent로 전파되지 않도록 scope를 제한할 수 있습니다.

18. CI/CD 연결

GitHub Actions 최소 CI 구조

# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main, develop]

jobs:
  quality-gate:
    name: Quality Gate
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Lint
        run: npm run lint

      - name: Typecheck
        run: npm run typecheck

      - name: Unit Tests
        run: npm run test -- --coverage

      - name: Build
        run: npm run build

      - name: Security Audit
        run: npm audit --audit-level=high
        continue-on-error: true

19. Auto Memory 활용

(출처: Anthropic Memory Docs)

Claude가 세션 중 스스로 메모를 저장하는 시스템입니다. 빌드 명령어, 디버깅 패턴, 아키텍처 노트, 코드 스타일 선호도 등이 자동으로 축적됩니다.

Auto Memory 저장 위치

~/.claude/projects/<project-path>/memory/
  ├── MEMORY.md        # 인덱스 파일 (최대 200줄 또는 25KB, 세션 시작 시 자동 로드)
  ├── debugging.md     # 디버깅 관련 학습
  ├── architecture.md  # 아키텍처 결정 메모
  └── preferences.md   # 코드 스타일 선호도
# Claude Code 내에서 확인 및 편집
/memory

Subagent Memory: Subagent에 memory: project를 설정하면 .claude/agent-memory/<agent-name>/에 누적 메모리가 쌓입니다. 이 디렉터리는 버전 관리가 가능하여 팀 전체가 공유할 수 있습니다.
