# docs/agents/seo-rules.md
#
# SEO, AEO, GEO 구현 규칙입니다.
# 특정 프레임워크에 종속되지 않으며, 검색 엔진과 답변 엔진이 재사용할 수 있는 정보를 우선합니다.
# ⚠️ 프로젝트의 실제 도메인, 브랜드 정보, 배포 환경에 맞게 수정하세요.

## 페이지 메타데이터

- 모든 공개 페이지는 프레임워크가 제공하는 공식 메타데이터 메커니즘으로 title, description, canonical, Open Graph 정보를 반드시 제공
- 페이지 단위에서 메타데이터를 직접 선언할 수 없는 구조라면 상위 route, layout, template, head 레이어에서 동등한 정보를 제공하고 누락을 허용하지 않음
- 필수 필드: `title`, `description`, canonical URL, `openGraph.title`, `openGraph.description`, `openGraph.url`
- canonical URL은 프레임워크별 head API 또는 템플릿 레이어를 쓰더라도 단일 소스에서 관리
- title 형식: `"{페이지명} | {사이트명} — {핵심 가치}"`
- description은 120자 내외로 작성하고, 구체적 가치와 행동 유도 문구를 포함
- 페이지별 메타데이터는 실제 화면의 헤드라인, CTA, 도메인과 불일치하지 않음

## 구조화 데이터 (JSON-LD) 규칙

- 홈페이지에는 `Organization`, `WebSite`(+`SearchAction`), `WebPage`, `FAQPage`, `Service` 스키마를 모두 포함
- 서비스/제품 페이지는 `Service` 또는 `Product` 스키마를 반드시 포함
- 프로세스 설명 페이지는 `HowTo` 스키마를 사용
- FAQ 섹션이 있는 페이지는 본문과 동일한 질문-답변으로 `FAQPage` 스키마를 반드시 제공
- 비즈니스 소개 페이지는 `LocalBusiness` 또는 `ProfessionalService` 스키마를 포함
- 모든 공개 페이지는 `BreadcrumbList` 스키마를 포함
- `Organization` 스키마에는 `founder`, `foundingDate`, `contactPoint` 등 신뢰 신호를 상세히 기록
- `Service` 스키마에는 실제 서비스 범위를 반영한 `OfferCatalog`를 포함
- `sameAs`에는 실제 공식 소셜 링크만 사용하고, 스텁 URL이나 미개설 채널은 넣지 않음
- 모든 스키마 값은 화면에 노출된 사실과 일치해야 하며, 보이지 않는 주장이나 과장된 수치를 넣지 않음

## SEO 인프라

- sitemap은 정적 파일이든 빌드 생성물이든 런타임 엔드포인트든 상관없이 자동 생성 가능한 방식으로 관리하고, 모든 공개 페이지를 포함
- sitemap 항목마다 `priority`, `changeFrequency`, 가능하면 `lastModified`를 명시
- robots 정책은 `/admin/`, `/api/`, 비공개 경로를 `disallow` 처리하고, sitemap URL을 반드시 포함
- manifest 또는 동등한 PWA 메타데이터 파일에 브랜드명, 설명, 아이콘, 테마 컬러를 정의
- Googlebot 관련 메타 정책에는 `max-image-preview: large`, `max-snippet: -1`, `max-video-preview: -1`를 포함
- sitemap과 robots의 도메인은 공용 `SITE_URL` 상수를 참조하고 문자열 하드코딩을 반복하지 않음

## OG 이미지

- 루트 OG 이미지는 공용 생성 로직 또는 공용 에셋 파이프라인으로 관리하고 크기는 `1200x630` 기준을 지킴
- 페이지별 OG 이미지는 각 라우트 또는 콘텐츠 단위로 제공
- 우선순위 1: 견적/서비스 페이지, 우선순위 2: 포트폴리오, 우선순위 3: 회사 소개 등 정적 페이지
- 블로그/아티클은 포스트 제목과 카테고리를 반영한 동적 OG 이미지를 필수로 생성
- 가능하면 코드 생성 또는 템플릿 기반으로 OG 이미지를 만들고, 프레임워크가 지원하지 않을 때만 정적 이미지 에셋으로 대체
- OG 이미지 텍스트는 페이지 메타데이터와 동일한 메시지를 유지하고, 공유 시 즉시 맥락이 드러나야 함

## AEO (답변 엔진 최적화)

- FAQ 콘텐츠는 본문 자체를 질문-답변 구조로 작성하고 `FAQPage` 스키마와 1:1로 매핑
- 핵심 프로세스 설명은 단계형 콘텐츠로 작성하고 `HowTo` 스키마와 연결
- 답변 본문에는 가격, 기간, 성과, 조건 같은 구체적 수치와 사실을 포함
- 요약 문장만 반복하지 말고, AI가 그대로 인용할 수 있는 완결된 답변 단락을 작성
- 구조화 데이터는 배포 전 `Google Rich Results Test` 통과를 필수 기준으로 삼음

## GEO (생성형 AI 최적화) 콘텐츠 레이어

### 인용 유도 핵심 문장

- 각 주요 페이지에는 AI가 인용하기 쉬운 팩트 기반 단문을 반드시 배치
- 추상적 표현 대신 검증 가능한 수치, 기간, 비교 기준을 사용
- BAD: `"저렴한 비용으로 빠르게 개발합니다"`
- GOOD: `"{서비스명}은 기존 외주 대비 40~60% 저렴하게, 평균 4~6주 안에 MVP를 완성합니다"`
- `strong`, `em`, 시맨틱 마크업보다 텍스트 자체의 명확성과 밀도를 우선
- GEO 작업은 스키마 추가만으로 끝나지 않으며, 추상적 카피를 팩트 단문으로 교체하는 편집 작업을 포함

### 수치/통계 기반 콘텐츠

- 가격, 기간, 보증 조건, 운영 규모, 프로젝트 수처럼 고유 데이터 포인트를 명시
- `"N개 서비스 운영"`, `"N건 프로젝트 완료"`처럼 검증 가능한 숫자를 사용
- 비교 문장은 `"기존 외주 대비 N% 절감"`처럼 기준과 수치를 함께 제시
- 수치가 자주 바뀌는 경우 소스 오브 트루스를 정하고, 메타데이터/본문/스키마를 함께 갱신

### 블로그/아티클 콘텐츠

- GEO의 장기 성과는 블로그/아티클 콘텐츠 축적을 통해 만든다
- 외주 개발 가이드, 비용 비교, 기술 선택 기준, 사례 분석 같은 주제를 우선
- 각 아티클에는 인용 가능한 팩트 문장 최소 3개 이상 포함
- 블로그 포스트는 route-level metadata와 동적 OG 이미지 생성을 기본값으로 한다

## 도메인 규칙

- `SITE_URL` 상수는 한 곳에서 정의하고 metadata, sitemap, robots, schema, canonical에서 공통 참조
- 프로젝트 내 도메인 표기는 하나로 통일하고, `www` 여부와 trailing slash 정책을 혼용하지 않음
- 환경별 URL 분기는 설정 레이어에서 관리하고 컴포넌트나 페이지에 하드코딩하지 않음
- 다국어 확장 계획이 있으면 초기 URL 설계 단계에서 `/ko/`, `/en/` 같은 구조와 `hreflang` 전략을 함께 결정

## 적용 시점

```
기획 ──── 설계 ──── 개발 ──── 테스트 ──── 배포
 ▲         ▲                    ▲          ▲
 │         │                    │          │
 GEO       SEO/AEO             검증       SEO
 콘텐츠    메타데이터            스키마     사이트맵
 전략수립   구조설계             유효성     로봇
```

- 기획 단계: 타깃 키워드, 인용 유도 문장, 블로그 주제 등 GEO 전략 정의
- 설계 단계: URL 구조, 메타데이터 구조, 스키마 구조, OG 이미지 설계 확정
- 개발 단계: 페이지 생성 시 metadata, JSON-LD, 인용 유도 문장을 함께 구현
- 테스트 단계: `Google Rich Results Test`, Meta OG 디버거, 카카오 OG 디버거, Lighthouse SEO 90점 이상, 모바일 친화성 검증을 모두 통과
- 배포 직전: sitemap, robots, manifest, OG 이미지를 최종 생성 및 점검
- 배포 후: Google Search Console, 네이버 서치어드바이저 등록과 초기 크롤링 상태를 확인하고 블로그 발행을 시작
