# 2026년 4월 17일 — 하네스(Harness) 개선 요약

**대상 독자**: 비개발자(기획, 디자이너, 관리자 등) — 기술 용어는 가능한 한 쉬운 말로 설명했습니다.

---

## 한 줄 요약

"AI가 빠르고 안전하게 일하도록 도와주는 장비(하네스)"를 크게 손봤습니다. 이제 개발 한 건(story)을 처리하는 시간이 **5~10분 → 1분 이내**로 단축되고, 작업 중 실수나 소실이 생겨도 **되돌릴 수 있는 안전망**이 이중삼중으로 깔렸습니다.

---

## 배경: 하네스(Harness)가 뭔가요?

말(馬)에 씌우는 "마구"에서 온 단어입니다. AI 개발 도구(Claude Code, Codex 등)는 아주 똑똑한 말과 같아서, 그냥 풀어놓으면 엉뚱한 방향으로 달려갑니다. 하네스는 그 말이 **올바른 방향으로, 안전하게, 일정한 속도로** 일하도록 잡아주는 보조장비입니다.

구체적으로는 이런 것들이 포함됩니다:
- AI가 파일을 고칠 때마다 "문법이 맞는지" 자동 점검하는 장치
- AI가 끝낸 작업이 정말 잘 돌아가는지 확인하는 검증 스크립트
- AI가 위험한 명령(예: 전체 파일 삭제)을 내리려 하면 차단하는 장치
- 팀의 규칙(코딩 스타일, 보안, 성능 등)을 AI가 읽고 따르도록 하는 규칙 파일

---

## 무엇이 문제였나 (1개월 사용 후 발견된 5가지)

1. **개발 하나 하는 데 너무 오래 걸렸다** (story 1건당 5~10분, 3건에 2시간 이상)
2. **Windows에서 검증을 돌리면 엉뚱한 프로그램(Docker)이 자동 설치**되거나 오류가 나는 경우가 있었다
3. **회사 표준 배포 흐름(`develop → main → 자동 배포`)이 하네스에 반영되어 있지 않았다** — 하네스는 바로 `main`에 합치도록 설계돼 있었음
4. **git commit(작업 기록 저장)을 놓쳐서 복구 불가능한 상황**이 생길 수 있었다
5. **같은 검증을 3번이나 돌리는데, 각자 고유 가치 없이 그냥 중복**처럼 보였다

---

## 어떻게 바꿨나 (7가지 조치 + 복구 안전망)

### ① Story 검증이 1분 안에 끝나도록 개선

**원인**: 검증 스크립트가 "관련 파일만 테스트"를 실패하면 조용히 **전체 파일 테스트로 전환**되고 있었습니다. 이 과정이 겉으로 보이지 않아서 "검증이 왜 이렇게 느리지?"만 남았습니다.

**수정**: 이 "조용한 전환"을 제거했습니다. 관련 테스트만 돌리거나, 도구가 없으면 명시적 에러로 알립니다. 예상 시간: **5~10분 → 30초~1분**.

### ② AI가 파일을 고칠 때마다 돌던 무거운 검사를 가볍게

**원인**: AI가 파일 하나 고칠 때마다 **프로젝트 전체를 검사**하고 있었습니다. 한 작업(story)에서 파일 15개 고치면 같은 검사를 15번 반복 — 5~15분 낭비.

**수정**: 고친 그 파일 하나만 검사합니다. 전체 검사는 작업이 끝날 때 1번만. 예상 시간: **파일당 10~60초 → 5초 이내**.

### ③ 작업 기록(git commit)을 놓치지 않도록 3중 안전망

1. **커밋 형식 자동 검증**: 메시지 형식이 정해진 규칙을 따르는지 자동 확인. 엉터리 메시지 차단.
2. **세션 종료 경고**: AI 세션이 끝날 때 "저장 안 된 변경이 있습니다!"라고 알려줌 (차단은 아님).
3. **AI 친화 메시지 규칙**: 커밋 메시지에 "어떤 파일을 / 왜 바꿨는지"를 구조화해서 적도록 권장. 나중에 AI가 과거 기록을 뒤질 때 찾기 쉬워집니다.

추가로 `.env` 같은 **비밀 파일을 실수로 기록하려 하면 자동 차단**합니다.

### ④ 회사 표준 배포 흐름을 하네스에 반영

회사 표준: `개발 완료 → develop 브랜치 → 자동 검증 → main 브랜치 → 사내 서버 자동 배포`

하네스가 이제 이 흐름을 그대로 따릅니다. AI가 develop에 올리면 GitHub가 자동 검증하고, main까지 가면 사내 Docker 서버에 배포됩니다.

### ⑤ 사라진 브랜치 정리를 "회고 시점"으로 옮김

**이전**: 매주 자동으로 브랜치를 청소하는 시스템이 있었지만, 사람이 모르는 사이에 작업이 사라질 위험이 있었습니다.

**수정**: 회고(Phase C) 때 사람이 확인하면서 정리합니다. **중요**: 삭제 전에 항상 `archive/<이름>/<날짜>` 형태의 "**복구 태그**"를 만들어 원격에 보관합니다. 나중에 되살릴 수 있습니다. 실수로 지워도 안전.

복구 방법 예시:
```
git checkout -b 되살린브랜치 archive/story_로그인화면/20260417
```

### ⑥ Windows 환경에서 엉뚱한 프로그램이 설치되는 문제 차단

**원인**: Windows에서 검증을 돌릴 때 "bash"라는 이름만 보고 **WSL**(리눅스 환경)을 쓰는 바람에 Docker가 자동 설치되거나 오류가 나고 있었습니다.

**수정**: Windows에서는 이제 **Git Bash**(Git 설치 시 같이 깔리는 가벼운 셸)만 사용합니다. WSL을 꼭 쓰려면 환경변수로 명시적 허용해야 합니다. 부작용 차단.

### ⑦ 검증 3번이 각자 다른 역할을 하도록 재설계

**이전**: Story 검증, Epic 검증, CI(원격) 검증이 거의 똑같은 걸 돌려 "중복 낭비"처럼 보였습니다.

**수정**: 4층 구조로 재설계. 각 층마다 고유한 역할:

| 시점 | 목적 | 고유 가치 |
|---|---|---|
| Story 완료 (1분 이내) | 빠른 피드백 | 이 작업에서만 바뀐 파일 |
| Epic 완료 (3~5분) | 통합 점검 | 하네스만 할 수 있는 특수 체크(보안 패턴 등) |
| develop 올릴 때 (5~8분) | 깨끗한 환경 재검증 | 커버리지, 보안 감사, Docker 이미지 만들기 |
| main 올릴 때 (5~10분) | 프로덕션 배포 | 사내 서버에 실제 배포 + 복구용 백업 |

중복이 아닌 "방어 심층화(Defense in Depth)" — 여러 겹의 안전망.

## 오늘 작업한 파일 목록 (기술자용 참고)

커밋 11개가 main에 합쳐졌습니다. 간단히:

- `.github/workflows/ci.yml`, `deploy.yml`: develop 트리거 + 배포 분리
- `docs/agents/testing-rules.md`, `workflow-rules.md`, `coding-rules.md`: 4층 검증 + 커밋 규칙
- `README.md`, `CLAUDE.md`: develop 흐름 반영

## 앞으로 해야 할 일 (사용자 쪽)

1. 이 템플릿을 다른 프로젝트에 복사한 후 **1회만 실행**:
   ```
   ./scripts/setup/install-git-hooks.sh        (Mac/Linux)
   ./scripts/setup/install-git-hooks.ps1       (Windows)
   ```
2. GitHub 저장소 설정에서 **Secrets 4개 등록** (배포용 SSH 정보 — `DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_SSH_KEY`, `DEPLOY_APP_DIR`)
3. 실제 프로젝트에 `develop` 브랜치 생성: `git checkout -b develop && git push -u origin develop`

이 3가지만 해두면 이후로는 하네스가 알아서 돌아갑니다.
